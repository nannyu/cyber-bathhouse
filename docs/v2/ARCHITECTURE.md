# 系统架构设计

## 1. 架构原则

**服务器权威（Server Authoritative）**

```
Client / Agent 只提交意图（Intent）
    ↓
Game Server 验证合法性
    ↓
Game Server 更新世界状态
    ↓
Game Server 广播结果给所有客户端
```

不允许客户端直接修改游戏状态。所有状态变更必须经过服务端验证。

## 3. 核心模块职责

### 3.1 RoomManager

- 创建/销毁房间
- 管理房间内玩家和 Agent 列表
- 房间状态快照和恢复
- 房间配置（模式/人数上限/地图）

### 3.2 GameLoop

- 固定 Tick 频率（10Hz，每 100ms 一次）
- 每 Tick：更新设施状态 → 处理顾客 → 结算资源 → 检查事件 → 广播状态
- 日夜循环推进

### 3.3 AgentTick

- 每 N 个 GameTick 向 Agent 推送感知状态（默认每 2 秒）
- 接收 Agent 返回的动作列表
- 动作排队等待 GameLoop 处理
- 超时处理（默认 wait）

### 3.4 ActionValidator

- 校验动作合法性（位置可达？资源足够？冷却完毕？）
- 统一处理人类玩家和 Agent 的动作
- 记录审计日志

## 4. 数据流

### 4.1 人类玩家操作流

```
玩家点击地图 → Client 发送 Intent → Server 验证 → 更新状态 → 广播
```

### 4.2 Agent 操作流

```
Server 推送 Perception → Agent 决策 → Agent 返回 Actions → Server 验证 → 更新状态 → 广播
```

### 4.3 状态同步流

```
GameLoop Tick → 计算新状态 → 生成 Delta → 广播给所有 Client
                                        → 记录日志
                                        → 触发事件检查
```

## 5. 房间生命周期

```
创建房间 → 等待加入 → 游戏开始（日循环）→ 日结算 → 继续/结束
                ↑                              │
                └──────── 新一天 ───────────────┘
```

## 6. 通信协议

### 6.1 Client ↔ Server（WebSocket）

| 方向 | 事件 | 数据 |
|------|------|------|
| C→S | `action` | `{ type, target, params }` |
| C→S | `chat` | `{ channel, text }` |
| S→C | `state:update` | 房间状态 Delta |
| S→C | `event:triggered` | 事件数据 |
| S→C | `task:assigned` | 任务数据 |
| S→C | `day:end` | 日结算报告 |

### 6.2 Server → Agent（CBAP HTTP）

| 方向 | 端点 | 数据 |
|------|------|------|
| S→A | `POST /agent/tick` | Perception 状态 |
| A→S | Response | Actions 列表 |

### 6.3 Server → Agent（CBAP WebSocket）

| 方向 | 事件 | 数据 |
|------|------|------|
| S→A | `perception` | 感知状态 |
| A→S | `actions` | 动作列表 |

---

## 7. 安全边界

| 层级 | 措施 |
|------|------|
| 网关 | Rate limit、Token 验证、IP 限制 |
| 动作 | Schema 校验、权限检查、冷却检查 |
| 状态 | 最小暴露原则（Agent 只看到视野内） |
| 日志 | 全量记录输入/输出/结果 |
| 隔离 | 房间级状态隔离，跨房间不可访问 |
