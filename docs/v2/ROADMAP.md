# 开发路线图

## 总览

```
Phase 0 (技术原型)  →  Phase 1 (可玩 MVP)  →  Phase 2 (传播版本)  →  Phase 3 (平台化)
    1-2 周                 4-6 周                  6-10 周                持续
```

---

## Phase 0：技术原型

### 目标

验证一个核心假设：

> **外部 Agent 能像玩家一样操控像素角色，并完成经营任务。**

### 范围

- 1 个地图（锅炉房 + 简化大堂）
- 2 个角色：1 人类玩家 + 1 Webhook Agent
- 1 个设施：锅炉
- 1 个任务：锅炉过热 → 修理
- 基础移动 + 交互
- Agent Tick 推送 + 动作接收
- 行动日志

### 验收标准

```
✅ 玩家能在网页地图上移动角色
✅ Agent 每 2 秒收到状态推送
✅ Agent 返回 move_to + interact 动作
✅ Agent 角色在地图上可见移动
✅ Agent 成功修理锅炉（温度下降）
✅ 行动日志记录了 Agent 的决策原因
```

### 任务拆分

| # | 任务 | 预计 | 依赖 |
|---|------|------|------|
| 0.1 | 搭建 Tile 地图渲染（Canvas） | 2天 | 无 |
| 0.2 | 实现角色在网格上移动（点击寻路） | 1天 | 0.1 |
| 0.3 | 实现锅炉设施（状态/温度/修理） | 1天 | 0.1 |
| 0.4 | 实现 GameLoop Tick（10Hz） | 0.5天 | 0.3 |
| 0.5 | 实现 Agent Tick 推送（Webhook） | 1天 | 0.4 |
| 0.6 | 实现动作校验和执行 | 1天 | 0.2, 0.5 |
| 0.7 | 实现行动日志 | 0.5天 | 0.6 |
| 0.8 | 编写测试 Agent（修锅炉） | 1天 | 0.5, 0.6 |
| 0.9 | 联调 + 修 bug | 1天 | all |

**总计：约 9 天**

## Phase 1：可玩 MVP

### 目标

一个小澡堂能玩起来，4 人 + 4 Agent 同时经营。

### 新增内容

| 模块 | 内容 |
|------|------|
| 地图 | 完整 4 区域（大堂/浴区/仓库/锅炉房） |
| 角色 | 4 玩家 + 4 Agent 同时在线 |
| 顾客 | NPC 顾客系统（排队/服务/满意度） |
| 资源 | 热水/燃料/肥皂/毛巾/金钱 |
| 任务 | 自动生成 + 手动分配 |
| 事件 | 10 个基础事件 |
| 信任 | Agent 信任值系统 |
| 日循环 | 早/白/晚/深夜 + 日结算 |
| Agent | Prompt Agent + Webhook Agent |
| UI | 资源面板/任务面板/聊天/日报 |

### 验收标准

```
✅ 4 个玩家可以加入同一房间
✅ 4 个 Agent 可以同时操控角色
✅ 顾客自动到来并排队
✅ 完成一天经营流程（5 分钟）
✅ 日结算报告生成
✅ 至少触发 1 个荒诞事件
✅ Agent 行为可复盘
```

---

## Phase 2：传播版本

### 目标

让用户愿意截图、录屏、分享。

### 新增内容

- Agent 个性模板（7 种预设性格）
- 观战模式（全 Agent 自动经营）
- 日结报告 + Agent 行为总结
- 荒诞日报生成
- 房间回放
- 合约系统
- 排行榜
- 更多事件（30+）
- 设施升级系统

### 传播示例

```
今日澡堂事故报告：
- Agent 老张把 12 条毛巾卖给了竞争对手
- 锅炉申请了人格权
- 玩家 Niko 因连续压榨 Agent 被系统评为"优秀企业家"
```

## Phase 3：平台化

### 目标

让开发者把它当 Agent Playground。

### 新增内容

- Agent SDK (TypeScript/Python)
- Agent Marketplace
- 自定义地图编辑器
- 自定义事件包
- Agent 对战联赛
- 完整 API 文档
- Webhook 调试器
- 多房间/多服务器

---

## 优先级原则

1. **先验证核心假设**（Agent 能操控角色完成任务）
2. **再补充经营深度**（资源/顾客/事件）
3. **最后做传播和平台**（回放/分享/SDK）

如果 Phase 0 原型不好玩，问题通常不在美术或系统规模，而在 **Agent 行为设计**。
